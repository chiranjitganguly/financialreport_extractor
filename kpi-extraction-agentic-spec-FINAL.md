# KPI Extraction Agentic Workflow — Build Specification

**Status**: Final for v1 build. Technology-agnostic — LLM provider, vector DB, document DB, and orchestration framework are to be selected separately; nothing below assumes a specific one.

Companion diagram: `kpi_extraction_workflow.drawio` (open in [diagrams.net](https://app.diagrams.net) / draw.io).

---

## 1. Overview

This system extracts a defined set of financial KPIs from a report (annual report, quarterly report, or regulatory filing), reading narrative text, tables, charts/graphs, and footnotes, and relating them to each other (e.g. a table cell to the footnote that qualifies it, or a chart to the KPI it depicts). It resolves each KPI through a three-tier cascade (deterministic → semantic → LLM), validates the results against accounting consistency rules, retries correctable failures within a bounded budget, and produces a final report that explicitly separates resolved values from anything that still needs a human's judgment.

**Pipeline at a glance**:

```
Ingestion & Classification (1)  ─┐
Multimodal Element Extraction(1b)─┴─→ Section Parser & Splitter (2) → Persistence & Indexing (3)
                                                                              │
                              ┌───────────────────────────────────────────────┘
                              ▼
            Tier 1: Deterministic (4) → Tier 2: Semantic (5) → Tier 3: LLM (6)
                              │                                       │
                              └──────────► Cross-Section Discrepancy Resolution (6a)
                                                       │
                                                       ▼
                                              Validator (7) ⇄ Retry Controller (8)
                                                       │
                                                       ▼
                                            Consolidation (9) → Final JSON
```

## 2. Shared Data Contracts

These are the stable interfaces every agent reads or writes. Treat them as the wire format of the system.

### `ReportMetadata`
```json
{
  "report_id": "string",
  "report_type": "annual_report | quarterly_report | regulatory_filing",
  "language": "string (ISO code)",
  "accounting_standard": "string (e.g. IFRS, US-GAAP, IND-AS)",
  "industry": "string",
  "fiscal_year": "string — granularity depends on report_type: financial year for annual_report (e.g. 'FY2025'); specific quarter for quarterly_report (e.g. 'Q2 FY2025'); whichever period is covered for regulatory_filing",
  "country": "string | null — reporting entity's country of incorporation/listing; sourced from the company reference map in Agent 1, no fallback classification defined for it (low priority signal — added for potential future jurisdiction-specific KPI/disclosure logic, not currently consumed by any other agent)"
}
```

### `TaxonomyEntry`
One row in the static Taxonomy Map — externally maintained, not generated by the pipeline.
```json
{
  "kpi_id": "string",
  "kpi_name": "string",
  "definition": "string — plain-language definition of what this KPI means and how it should be computed/identified, used by Tiers 1–3 and the Validator to disambiguate near-duplicate labels and to ground LLM extraction prompts",
  "canonical_sections": ["string"],
  "applicable_industries": ["string"],
  "applicable_report_types": ["string"],
  "applicable_accounting_standards": ["string"],
  "aliases": ["string"]
}
```
**Rules**:
- A KPI may legitimately appear in more than one section (`canonical_sections` is a list) — e.g. summarized in MD&A and detailed in the financial statement notes. All listed sections are checked; disagreement is handled by §8 (Cross-Section Discrepancy Resolution).
- An empty/missing `applicable_industries`, `applicable_report_types`, or `applicable_accounting_standards` means the KPI applies to **all** values for that dimension. No explicit `"ALL"` literal is needed — taxonomy authors only populate a field to *restrict* a KPI to a subset.
- `definition` is mandatory for every entry. It is not just documentation — it is fed directly into Tier 3's extraction prompt and into the Validator's footnote-materiality judgment, so it should be written as a working instruction ("Net revenue from primary operations, excluding non-operating income"), not a one-line gloss.

### `TableElement`
```json
{
  "table_id": "string",
  "caption": "string | null",
  "section_name_canonical": "string",
  "page": "number",
  "rows": [
    {
      "row_label": "string",
      "cells": [
        { "column_label": "string (e.g. 'FY2025', 'FY2024')", "value": "string | number", "footnote_refs": ["string"] }
      ]
    }
  ],
  "footnote_refs": ["string"]
}
```

### `ChartElement`
```json
{
  "chart_id": "string",
  "caption": "string | null",
  "section_name_canonical": "string",
  "page": "number",
  "image_ref": "string (storage pointer to the cropped chart image)",
  "interpretation": "string (structured description: chart type, axis labels, series, key data points)",
  "interpretation_confidence": "number (0-1)",
  "footnote_refs": ["string"]
}
```

### `FootnoteElement`
```json
{
  "footnote_id": "string",
  "marker": "string (e.g. '1', 'a', '*')",
  "text": "string",
  "section_name_canonical": "string",
  "page": "number",
  "anchors": [
    { "element_type": "table_cell | chart | text", "element_id": "string", "location": "string" }
  ]
}
```
**Scope (v1)**: anchoring is **marker-based only** — a literal footnote symbol attached to a table cell, chart caption, or text span, matched to the footnote text that defines it. Narrative cross-references with no marker (e.g. "as discussed in Note 5" in prose) are out of scope for v1.

### `Section`
```json
{
  "section_name_raw": "string",
  "section_name_canonical": "string (one of the Taxonomy Map's canonical section names, or 'OTHER' for sections with no KPI relevance — e.g. Chairman's Letter, Corporate Governance commentary)",
  "alignment_confidence": "number (0-1)",
  "alignment_source": "fuzzy_match | llm_fallback | best_guess_unresolved",
  "content_markdown": "string (narrative text only)",
  "tables": ["TableElement"],
  "charts": ["ChartElement"],
  "footnotes": ["FootnoteElement"],
  "page_range": [12, 15]
}
```
Structured `tables`/`charts`/`footnotes` are kept alongside `content_markdown`, not flattened into it — this is what allows real row/column lookups in Tier 1 and faithful structure in the Tier 3 prompt, rather than a lossy markdown approximation.

**Added in this revision**: `alignment_confidence` and `alignment_source` were missing from the original contract despite Agent 2's spec already describing low-confidence alignment handling — there was nowhere to actually carry that information downstream. `alignment_source: best_guess_unresolved` is the terminal state when a section's alignment retry budget is exhausted and the pipeline proceeds with its best guess anyway (per Agent 2's low-confidence handling) — any KPI later extracted from such a section should inherit a `needs_human_review` flag, since the section it came from was never confidently identified.

### `ExtractionRecord`
The unit every retrieval tier produces — the mutable "working ledger" entry, distinct from the static `TaxonomyEntry`.
```json
{
  "kpi_id": "string",
  "value": "string | number | null",
  "fiscal_year": "string",
  "section": "string",
  "page": "number | null",
  "method": "deterministic | semantic | llm",
  "source_element_type": "text | table_cell | chart | null",
  "footnotes": ["footnote_id, ..."],
  "confidence": "number (0-1)",
  "status": "found | not_found | flagged | needs_human_review",
  "review_reason": "low_confidence | validation_failed | not_found_after_retries | section_discrepancy | footnoted_caveat | null",
  "conflicting_values": [
    { "section": "string", "value": "string | number", "method": "deterministic | semantic | llm", "source_element_type": "text | table_cell | chart" }
  ],
  "attempts": [
    { "tier": "deterministic | semantic | llm", "value": "string | number | null", "confidence": "number (0-1)", "outcome": "found | not_found | flagged", "note": "string" }
  ]
}
```

**Field rules**:
- `source_element_type` records where the value physically came from. Table-cell values are generally more trustworthy than a number paraphrased from prose, and this field is required for the `conflicting_values` comparison.
- `footnotes` carries every footnote linked to the source cell/chart, regardless of materiality — Tiers 1–3 only record the link; the Validator (§9) judges whether it's substantive.
- `fiscal_year` must match `ReportMetadata.fiscal_year` — extraction must target the correct period/column, not an adjacent comparative-year column in the same table.
- `attempts` preserves full history (every tier's result), never overwritten — this is the audit trail.
- `status: needs_human_review` is the terminal state, reached via any of: retry budget exhausted while still `flagged`/`not_found`; confidence below threshold despite passing validation; a resolved cross-section discrepancy; or a material footnote caveat. `review_reason` records which.
- **Confidence is weighted equally across all three tiers** — no tier-based discount or boost. Tier 1 (deterministic, no natural probabilistic score) computes confidence as: `1.0` for an unambiguous single match, `0.5–0.7` when a heuristic had to tie-break multiple candidates.

### `ExtractionLedger`
A map of `kpi_id → ExtractionRecord`, initialized with every KPI from the taxonomy (filtered by this report's `industry` / `report_type` / `accounting_standard`) marked `not_found`. Every agent updates only the entries it resolves; this ledger is the shared mutable state passed across the whole pipeline.

## 3. Agents

### Agent 1 — Ingestion & Classification

**Purpose**: Normalize the raw input and establish report-level metadata before any KPI work begins.

**Input**: Raw file (PDF/DOCX/HTML/etc.), optional user-supplied report type.

**Sub-tasks**:
- Report Type Selector (use provided value if given, else classify: annual / quarterly / regulatory filing)
- Language Detector
- Accounting Standard Detector
- Industry Selector
- Document → Markdown Converter — produces the narrative text backbone with heading hierarchy and page boundaries preserved. Tables, charts, and footnotes are **not** flattened into this markdown; they're extracted separately by Agent 1b.

**Output**: `ReportMetadata` + narrative Markdown (with page anchors).

**Human-in-the-loop**: if confidence on any classification field falls below threshold, the report is queued in an `awaiting_input` state — **async, not a synchronous block**. The pipeline keeps processing other reports; this one resumes once a human corrects/confirms the field via the review interface.

---

### Agent 1b — Multimodal Element Extraction

**Purpose**: Extract tables, charts/graphs, and footnotes as first-class structured elements, and establish the literal cross-references that let later agents relate a number to its qualifying footnote, or a chart to the KPI it depicts.

**Input**: Raw file, `ReportMetadata`.

**Sub-tasks**:
- **Table extraction**: parse into structured `TableElement`s — row labels, column labels (typically fiscal periods, matched against `ReportMetadata.fiscal_year`), per-cell values, and footnote markers per cell.
- **Chart/graph extraction**: crop and store the image, generate a structured `interpretation` (chart type, axis labels, series, key values) via a vision-capable model call, and capture the caption/title (the strongest signal for which KPI it relates to).
- **Footnote detection & anchoring**: detect markers wherever they occur (table cell, chart caption, text) and the footnote text blocks that define them; link each occurrence via `FootnoteElement.anchors`.

**Output**: `List[TableElement]`, `List[ChartElement]`, `List[FootnoteElement]`.

**Why a separate agent**: isolates the most format-dependent, failure-prone work (table-parsing fidelity varies by source PDF quality; vision-based chart interpretation has its own error modes) so it can be built and iterated on independently of report-level classification.

---

### Agent 2 — Section Parser & Splitter

**Purpose**: Split the document into sections, align them to the Taxonomy Map's canonical vocabulary, and assign structured elements into their owning sections.

**Input**: Markdown document, `ReportMetadata`, the three element lists from Agent 1b.

**Sub-tasks**:
- Split on heading structure into raw sections.
- **Section alignment**: map each raw header to one or more canonical section names via fuzzy/semantic matching against the Taxonomy Map's vocabulary (handles e.g. "Statement of Profit & Loss" vs. "Income Statement"). Produces a per-section alignment confidence score.
- **Element assignment**: assign each table/chart/footnote to its owning section by page-range overlap.

**Output**: `List[Section]`.

**Low-confidence alignment handling**: retry the alignment itself (alternate matching strategy, lower similarity cutoff, or an LLM-assisted mapping attempt) using the **report's shared 2-turn retry budget** (see Agent 8 — this is not a separate local budget; an alignment retry here leaves only one turn for downstream validation retries). If still below threshold after the budget, proceed with the best-guess mapping (don't stall the pipeline), but any KPI later extracted from that section surfaces as `needs_human_review`.

---

### Agent 3 — Persistence & Indexing

**Purpose**: Persist parsed sections to both stores so later tiers can query them independently.

**Input**: `List[Section]`, `ReportMetadata`.

**Sub-tasks**:
- Persist each section to **Document DB**, keyed by `section_name_canonical` + `report_id`, including its `tables`, `charts`, and `footnotes` as structured sub-fields.
- Chunk and embed for **Vector DB**, metadata `{report_id, section_name_canonical, page_range, source_element_type}`, covering all content types:
  - Narrative text → standard text chunking
  - Tables → each row (or small group of related rows) serialized into embeddable text (e.g. `"Revenue — FY2025: 1,234; FY2024: 1,100"`)
  - Charts → the `interpretation` text is embedded
  - Footnotes → embedded independently, tagged with what they anchor to

**Output**: Storage references (document keys, vector collection/namespace id).

---

### Agent 4 — Tier 1: Deterministic Extraction

**Purpose**: Cheapest, highest-confidence extraction pass.

**Input**: Taxonomy entries filtered by `industry` + `report_type` + `accounting_standard`, `ExtractionLedger` (all `not_found`), Document DB.

**Logic**: For each taxonomy entry, fetch **all** its `canonical_sections`. Within each, check sources in order:
1. **Tables first** — match `row_label` against `kpi_name` + `aliases` (and `definition` for disambiguation); if matched, read the cell at the `column_label` matching `ReportMetadata.fiscal_year`. Set `source_element_type: table_cell`.
2. **Narrative text fallback** — pattern/regex match over `content_markdown` if no table match. Set `source_element_type: text`.

Record any linked footnote id(s) on the `ExtractionRecord` (materiality judged later, by the Validator). If a value is found in more than one candidate section, hand off to Cross-Section Discrepancy Resolution (§8).

**Output**: Updated ledger — resolved entries get `method: deterministic`; the rest flow to Tier 2.

---

### Agent 5 — Tier 2: Semantic Retrieval

**Purpose**: Catch KPIs deterministic matching missed due to phrasing variance.

**Input**: `ExtractionLedger` filtered to `not_found`, Vector DB.

**Logic**: Build a query from `kpi_name` + `aliases`, retrieve top-k chunks via vector search filtered by `canonical_sections` — searching across **all** indexed content types (narrative, table-row, chart-interpretation chunks), not just prose. Apply the **same deterministic/pattern logic as Tier 1**, scoped to the retrieved chunk(s). No LLM call here — the only difference from Tier 1 is *where* the pattern matching runs (a semantically-narrowed chunk vs. the whole section), not *how*.

**Output**: Updated ledger — resolved entries get `method: semantic`; remainder flows to Tier 3.

---

### Agent 6 — Tier 3: LLM Extraction

**Purpose**: Last-resort extraction using full section context and reasoning — and the only tier that can recover a KPI disclosed solely via a chart.

**Input**: `ExtractionLedger` filtered to `not_found`, full section content (narrative text + tables + charts + footnotes, not narrative alone) for each KPI's `canonical_sections`.

**Logic**: One batched prompt per section — list every unresolved KPI belonging to that section together (with `kpi_name`, `aliases`, `definition`), along with the section's serialized tables, footnote text, and chart content (either the `interpretation` text or the raw image as a vision input, depending on the chosen model — see Open Items). Request a single structured JSON response mapping each KPI to its value, page, `source_element_type`, and any qualifying footnote ids. Prompt instructions should be explicit per-KPI rather than one generic instruction, so accuracy doesn't degrade as batch size grows.

**Output**: Updated ledger — resolved entries get `method: llm`; any still unresolved are marked `status: not_found` (final — no further tiers).

---

### Step 6a — Cross-Section Discrepancy Resolution

**Purpose**: Resolve a KPI whose `canonical_sections` (or different source element types within one section — e.g. a table figure vs. a chart-derived figure) yielded disagreeing values during Tiers 1–3.

**Trigger**: Extraction succeeded more than once for the same KPI, across sections or element types, with disagreeing values.

**Logic**: Escalate to Tier 3 (LLM), even if a value was already found — give it all candidate values/sources and have it determine the authoritative one. **The LLM's value wins** and becomes the record's `value`; all other candidates are preserved in `conflicting_values`. Set `status: needs_human_review`, `review_reason: section_discrepancy`.

**Budget**: this is an immediate, one-shot escalation inline in the extraction cascade — it does **not** consume the shared 2-turn retry budget.

**Terminal status**: `section_discrepancy` never re-enters the Validator/Retry Controller loop, even if the resolved value later fails a tally check (that failure is logged in `attempts` for visibility, but doesn't trigger a retry). Rationale: a cross-source disagreement is a data-provenance problem — the document disagrees with itself — not a single-tier extraction error, so retrying extraction won't fix it; it goes straight to human judgment.

---

### Agent 7 — Validator

**Purpose**: Rule-based consistency check across the ledger, independent of which tier produced each value — plus footnote materiality classification.

**Input**: `ExtractionLedger` (post Tiers 1–3 and §8), externally-supplied **Validation Rules Map**.

**Logic**:
- **Tally rules**: cross-field checks (e.g. Net Income = Revenue − Expenses; subtotal = sum of line items; period-over-period plausibility). On failure, mark **all participating KPIs** `status: flagged` (no fault-ranking — a 3-way tally failure flags and retries all 3 fields together), with an `attempts` entry per KPI noting the discrepancy.
- **Low-confidence pass-through**: entries that pass validation but whose `confidence` is below threshold are marked `status: needs_human_review`, `review_reason: low_confidence` — they skip the retry cascade (nothing's wrong, just not confident) and go straight to the review list.
- **Footnote materiality classification**: for every record with a non-empty `footnotes` array, classify whether the linked text substantively qualifies the value (adjusted/non-GAAP, restated, partial-period, excludes-X language) versus a non-substantive citation. (Keyword heuristic as the v1 default; LLM judgment call as the more robust fallback — see Open Items.) If material: `status: needs_human_review`, `review_reason: footnoted_caveat` — applies even to an otherwise high-confidence, validation-passing value, since this is a disclosure-completeness concern, not a correctness one. **Terminal, like `section_discrepancy`** — never re-enters the retry loop.

Validation rules are config, supplied externally — the Validator consumes them, it doesn't hardcode formulas.

**Output**: Validated ledger + `flagged` KPI ids (all rule participants) + `needs_human_review` ids (low-confidence and footnote-caveat).

---

### Agent 8 — Retry Controller

**Purpose**: Re-run extraction for flagged/unresolved KPIs within a bounded, report-level retry budget.

**Input**: Ledger with `flagged`/`not_found` entries, turn count so far.

**Logic**: Each turn re-invokes extraction for **all** currently-flagged KPIs together (not staggered individually), jumping straight to the tier most likely to fix the issue rather than re-running the full cascade:
- Tier 1 result flagged, looks like a section-matching/parsing error → retry via Tier 2 or Tier 3, not Tier 1 again.
- Tier 3 result flagged by a tally failure → re-run Tier 3 with the validator's specific discrepancy folded into the prompt (e.g. "previous value X should reconcile with Y — re-check this section").
- Never found by any tier → re-run Tier 3 with broader section context rather than looping 1→3 again.

**Budget**: **2 turns total per report** — a single shared counter, also drawn on by Agent 2's section-alignment self-correction (not two separate budgets). After 2 turns, anything still `flagged`/`not_found` is marked `status: needs_human_review` (`review_reason: validation_failed` or `not_found_after_retries`), `attempts` history intact.

**Output**: Updated ledger, looped back to the Validator if turns remain.

---

### Agent 9 — Consolidation

**Purpose**: Produce the final deliverable, with everything that needs human judgment explicitly surfaced.

**Input**: Final `ExtractionLedger`.

**Logic**: Partition by final `status`:
- `resolved` — `found`, confidence ≥ threshold, no validation flags, no unresolved discrepancy, no material footnote caveat
- `needs_review` — everything `needs_human_review`: validation failures, not-found-after-retries, low-confidence passes, section discrepancies, footnote caveats

**Output**:
```json
{
  "report_id": "string",
  "fiscal_year": "string",
  "resolved": [
    { "kpi": "string", "value": "...", "section": "string", "page": 14, "method": "deterministic", "source_element_type": "table_cell", "confidence": 0.93, "footnotes": [] }
  ],
  "needs_review": [
    {
      "kpi": "string",
      "value": "... | null",
      "section": "string",
      "page": "number | null",
      "source_element_type": "text | table_cell | chart | null",
      "confidence": 0.42,
      "review_reason": "low_confidence | validation_failed | not_found_after_retries | section_discrepancy | footnoted_caveat",
      "footnotes": ["..."],
      "conflicting_values": ["... populated only for section_discrepancy ..."],
      "attempts": ["... full history ..."]
    }
  ]
}
```
`resolved` entries still carry `footnotes` (typically empty) for transparency, without forcing non-material links into the review queue.

## 4. Orchestration Layer

A thin Orchestrator/Controller should:
- Own the `ExtractionLedger` as shared state across all agent calls.
- Sequence Agents 1→3 linearly, then run the Tier 1→2→3 cascade (with inline §8 discrepancy resolution where triggered), then the Validator ⇄ Retry Controller loop, then Consolidation.
- Track **one shared 2-turn retry budget per report**, spent by either Agent 2's alignment self-correction or the Validator/Retry Controller loop.
- Support an `awaiting_input` state for async human-in-the-loop classification review — other reports continue processing while one is queued.
- Treat `section_discrepancy` and `footnoted_caveat` as bypassing the Validator/Retry loop entirely — terminal on first detection.
- Decide routing on other failures (e.g. malformed JSON from Tier 3).

This is where you plug in whatever orchestration framework you choose (state machine, graph-based agent framework, or a simple sequential pipeline) — none of the agent boundaries above depend on that choice.

## 5. Open Items (decide before / during build)

- **Chart interpretation method**: pre-computed text description at ingestion (Agent 1b), or raw chart image passed as a vision input directly to the Tier 3 LLM call? The schema supports either — this is a cost/tech tradeoff for whichever LLM you select, not a design gap.
- **Footnote materiality classification method**: keyword heuristic (cheap, some false positives/negatives) vs. LLM judgment call (more robust, costs a call per footnoted value) — pick the v1 approach.
- **Table/chart-to-section assignment ambiguity**: an element straddling a section boundary has the same kind of uncertainty as section-header alignment, but no confidence/retry mechanism is defined for it yet — low priority unless source documents exhibit this often.
- Narrative-only footnote references (no marker) are explicitly **out of scope for v1** — confirm this is acceptable, since it means a footnote cited only in prose won't be auto-linked.

## 6. Design Decisions Log (for traceability)

Condensed record of the choices baked into this spec, in case a future change needs to know *why* something is the way it is:

- Retry budget is **report-level** (2 turns total, shared with section-alignment retries), not per-KPI — bounds cost.
- Confidence is **weighted equally** across all three tiers.
- A failed tally rule retries **all** participating KPIs, not a ranked subset.
- `fiscal_year` granularity follows `report_type` (year for annual, year+quarter for quarterly).
- Multi-section KPI disagreement resolves via **LLM-wins**, always flagged for review (`section_discrepancy`), and is **terminal** (no retry re-entry) even if it later fails validation.
- Section-alignment low confidence is surfaced for review only after self-correction attempts are exhausted from the shared budget.
- Tier 2 is vector search + the same deterministic logic as Tier 1, scoped to a chunk — no LLM call. Only Tier 3 calls an LLM.
- Validation rules and the Taxonomy Map are both externally maintained config, not hardcoded.
- Full per-KPI attempt history is retained for audit purposes — nothing is overwritten in place.
- HITL classification review is **async** (queue-and-resume), not a synchronous block.
- Footnote materiality (`footnoted_caveat`) is treated the same as `section_discrepancy`: terminal, never retried.
